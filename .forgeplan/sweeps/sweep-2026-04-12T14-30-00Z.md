## Findings

### F1 — Admin router double-prefix (all admin endpoints 404)
**Severity:** HIGH
**Node:** admin-surfaces
**Files:** `placementops/modules/admin/router.py`
**Agents:** adversary, contractualist, structuralist

`APIRouter(prefix="/api/v1")` in `admin/router.py` + `app.include_router(admin_router, prefix="/api/v1")` in `main.py` causes double-prefix. All admin routes resolve at `/api/v1/api/v1/admin/...`, `/api/v1/api/v1/templates/...`, etc. — returning 404 for all admin operations.

**Fix:** Remove `prefix="/api/v1"` from `APIRouter` in `admin/router.py`. Keep routes as `/admin/users`, `/templates/outreach`, `/imports`, `/admin/organization`, `/reference/*`. The `prefix="/api/v1"` in `main.py` is correct and must remain.

### F2 — Intake `POST /cases/{id}/assign` shadows clinical endpoint (unreachable)
**Severity:** HIGH
**Node:** intake-module
**Files:** `placementops/modules/intake/router.py`
**Agents:** structuralist, skeptic

Both `clinical/router.py` and `intake/router.py` define `POST /cases/{case_id}/assign`. FastAPI uses first-registered route wins. `clinical_router` is registered before `intake_router` in `main.py`, so the clinical handler always runs. The intake assign endpoint is unreachable dead code.

**Fix:** Rename intake's endpoint from `/cases/{case_id}/assign` to `/cases/{case_id}/assign-coordinator`. Update path in router decorator and in any tests referencing this path.

### F3 — API client sends unauthenticated requests (no Authorization header)
**Severity:** HIGH
**Node:** frontend-app-shell
**Files:** `frontend/src/client/index.ts`
**Agents:** adversary, pathfinder

`apiClient.fetch()` in `frontend/src/client/index.ts` does not attach `Authorization: Bearer <token>`. All requests from the frontend return HTTP 401. The Supabase client is available at `@/lib/supabase/client` but is not wired into the API client.

**Fix:** Inject the Supabase session token into the `fetch` helper — call `createClient().auth.getSession()` to retrieve the access token, then set `Authorization: Bearer ${session?.access_token}` in the headers.

### F4 — File upload reads full body before size check (OOM risk)
**Severity:** HIGH
**Node:** intake-module
**Files:** `placementops/modules/intake/router.py`
**Agents:** adversary, pathfinder, skeptic

`upload_import` reads the entire file body with `file_bytes = await file.read()` before checking `len(file_bytes) > MAX_UPLOAD_BYTES`. A malicious actor can send an arbitrarily large file, forcing the server to buffer it entirely in memory before the 413 check triggers.

**Fix:** Use `await file.read(MAX_UPLOAD_BYTES + 1)` to read at most one byte over the limit. If the returned buffer exceeds `MAX_UPLOAD_BYTES`, reject with 413. This caps memory usage at ~10 MB + 1 byte without reading the full payload.

### F5 — `patch_case` missing closed-case guard (terminal status mutation)
**Severity:** HIGH
**Node:** intake-module
**Files:** `placementops/modules/intake/service.py`
**Agents:** skeptic

`patch_case` in `intake/service.py` fetches the case and applies updates without checking `case.current_status`. A closed or placed case (`current_status in {"placed", "closed"}`) can be modified via `PATCH /cases/{id}`, violating the closed-case immutability rule enforced elsewhere in the system.

**Fix:** After fetching the case, add a check: if `case.current_status in {"placed", "closed"}`, raise `HTTPException(409, "Cannot modify a case in terminal status")`.

### F6 — `clinical_fit_score` returns 0.0 for zero-complexity patients (wrong penalty)
**Severity:** HIGH
**Node:** matching-module
**Files:** `placementops/modules/matching/engine.py`
**Agents:** skeptic

`compute_clinical_fit_score` returns `0.0` when `total_needed == 0` (patient has no clinical needs). This 0.0 is weighted at 30% of the overall score. A patient with no clinical needs receives a 30% penalty on ALL facilities, making them appear riskier than a patient with one matched clinical need. A zero-complexity patient should score all facilities equally at 1.0 on this dimension (no special needs = any facility is suitable).

**Fix:** Change `return 0.0` at line 496 to `return 1.0` in the `if total_needed == 0:` branch. Comment updated to: "No clinical complexity — patient is universally suitable → perfect score."

### F10 — LIKE metacharacter injection in patient name search
**Severity:** MEDIUM
**Node:** intake-module
**Files:** `placementops/modules/intake/service.py`
**Agents:** adversary

`list_cases` in `intake/service.py` line 416: `PatientCase.patient_name.ilike(f"%{search}%")` does not escape `%` and `_` metacharacters in the user-supplied `search` value. A search for `%_%_%_%_%` matches all patients. SQLAlchemy's `ilike()` prevents SQL injection but metacharacter matching exploits remain possible.

**Fix:** Escape LIKE metacharacters before interpolation:
```python
search_safe = search.replace("\\", "\\\\").replace("%", "\\%").replace("_", "\\_")
stmt = stmt.where(PatientCase.patient_name.ilike(f"%{search_safe}%", escape="\\"))
```
